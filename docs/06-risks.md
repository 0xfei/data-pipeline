## 2. P0 级风险

### R1：数据漂移未被发现

| 维度 | 内容 |
|---|---|
| **优先级** | 🔴 P0 |
| **说明** | 网页源内容劣化（如内容农场 SEO 优化、AI 生成内容增加）、GitHub 源质量下降（低质量 repo 涌入），未被漂移检测发现，劣质数据直接进入训练集，导致模型退化 |
| **触发条件** | 数据源的语义分布、质量分分布、语言分布发生统计显著变化 |
| **预警信号** | 周级 KS-test / MMD 检验 p-value < 0.01 连续 4 周；perplexity 分布中位数偏移 > 20%；各分类标签占比变化 > 10% |
| **缓解措施** | 1. 周级漂移检测（KS-test/MMD on embedding distribution + perplexity drift），Day 1 上线<br>2. 每两周自动生成漂移报告<br>3. 当预警触发时，暂停该数据源的数据进入训练集，人工排查<br>4. 建立历史 baseline 分布快照（每月一份）用于对比 |
| **R（执行）** | 数据团队：监控平台搭建 + 漂移检测算法实现 + 周级报告生成 |
| **A（负责）** | 数据链路负责人（你）：拍板是否暂停数据源、是否触发人工排查 |
| **C（咨询）** | 训练团队：漂移阈值多少算异常？对训练的实际影响评估 |
| **I（告知）** | 训练团队：月度漂移报告；法务：如漂移涉及合规数据 |

### R2：过拟合

| 维度 | 内容 |
|---|---|
| **优先级** | 🔴 P0 |
| **说明** | 数据配比不当导致模型对特定分布/来源/语言过拟合。典型表现：训练 loss 持续下降但下游 benchmark 不提升甚至下降；模型在某一数据源上表现极好，其他数据源表现差 |
| **触发条件** | 数据配比严重失衡（某来源 > 60%）、数据多样性不足、大量重复数据未被去重 |
| **预警信号** | 训练 loss 和验证 loss 背离 > 5%；benchmark 分数中某领域高、某领域异常低；去重后数据分布出现长尾集中的单一来源 |
| **缓解措施** | 1. 多维度配比动态调整（来源×质量×语言×类型）<br>2. 定期（每月）与训练团队复盘 loss/benchmark 趋势<br>3. 在数据配比时强制加入多样性约束（每个来源不低于 5%）<br>4. 去重阶段确保 MinHash+LSH 生效，消灭近重复数据<br>5. Phase 2 语义去重进一步消除功能等价重复 |
| **R（执行）** | 数据团队：多维度配比工具 + 数据分布可视化 + 去重质量保证 |
| **A（负责）** | 数据链路负责人（你）+ 训练团队负责人：联合决策配比调整 |
| **C（咨询）** | 训练团队：loss/benchmark 数据分析、配比需求 |
| **I（告知）** | 管理层：月度训练健康报告 |

### R3：高质量数据不足

| 维度 | 内容 |
|---|---|
| **优先级** | 🔴 P0 |
| **说明** | 经过粗过滤→细过滤→评测漏斗后，可用数据量远低于预期。可能原因：原始数据质量过差、过滤阈值过严、特定领域（如数学、逻辑推理）天然数据稀缺 |
| **触发条件** | 过滤后可用数据量 < 预期的 60% |
| **预警信号** | 各层过滤留存率连续 2 周低于阈值（粗过滤留存 < 70%、评测 L1 留存 < 70%、L2 quality_score > 40 占比 < 50%） |
| **缓解措施** | 1. Phase 1 就建立数据量预测模型，按源/语言/类型预测过滤后产出<br>2. 多渠道并行采集（加速 GitHub + Web + API 采集）<br>3. 合成数据补充（模板生成 + LLM 开放生成）<br>4. 冷门领域定向挖掘（如 arxiv 数学论文、LeetCode 题解、Stack Overflow）<br>5. 调整过滤阈值（如 MinHash Jaccard 从 0.8 放宽到 0.7） |
| **R（执行）** | 数据团队：数据量监控 + 多渠道加速采集 + 合成数据生成 |
| **A（负责）** | 数据链路负责人（你）：决定是否调整过滤阈值、是否扩大采集范围 |
| **C（咨询）** | 训练团队：最少需要多少数据？哪些领域的数据最稀缺？ |
| **I（告知）** | 管理层：如数据缺口 > 50%，需要提前预警预算和资源需求 |

## 3. P1 级风险

### R4：Embedding 成本失控

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 P1 |
| **说明** | 100PB 数据即使轻量模型也可能吃掉大量 GPU。实测吞吐可能远低于预估（复杂文本、长文档导致单 doc 处理时间增加），导致 GPU 费用超支 |
| **触发条件** | Phase 1 benchmark 实测吞吐 < 预估的 50%（即 < 3000 docs/s per A100） |
| **预警信号** | 单卡 throughput < 3000 docs/s（multilingual-e5-small）；单 batch 处理时间标准差 > 50%（数据不均匀） |
| **缓解措施** | 1. Phase 1 必须先做小规模 benchmark（1TB 数据）实测吞吐和成本<br>2. 如吞吐不达标：ONNX 量化模型、增大 batch size、换更轻量的模型（384d → 256d）<br>3. 如成本超预算：先对高 importance 数据做 embedding，低 importance 数据降级（仅存文本，不建索引）<br>4. 定期 （每周）监控 GPU 利用率和吞吐，设立成本消耗告警线 |
| **R（执行）** | 数据团队：benchmark 测试 + 模型优化 + 成本监控 |
| **A（负责）** | 数据链路负责人（你）：拍板模型替换或范围收缩 |
| **C（咨询）** | 平台团队：GPU 集群规格和成本优化经验 |
| **I（告知）** | 训练团队：embedding 精度下降时需同步 |

### R5：Spark ↔ Ray 数据流转瓶颈

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 P1 |
| **说明** | Lance 格式在 Spark 和 Ray 之间的读写性能未经验证。Spark 写 Parquet、Ray 读 Lance 是两套体系，可能存在格式转换开销或性能衰减 |
| **触发条件** | Lance 读写 benchmark 延迟超过预期（单 partition > 10s） |
| **预警信号** | 单 partition Lance 读写延迟 > 10s；格式转换时间 > 总处理时间的 20% |
| **缓解措施** | 1. Phase 1 必须做 Lance 格式的读写 benchmark（Spark 读 Parquet → Ray 读 Lance → Ray 写 Lance → Spark 读 Lance）<br>2. 统一文件大小（256-512MB），避免碎片<br>3. 如果 Lance 读写在 Spark 中性能差，改用 Parquet 作为中间格式，Lance 仅用于 Ray 侧<br>4. 考虑使用 Ray Data 直接读 Iceberg 表（跳过 Spark 读 Lance 的需求） |
| **R（执行）** | 数据团队：benchmark + 格式优化 |
| **A（负责）** | 数据链路负责人（你） |
| **C（咨询）** | Lance 社区 / 平台团队 |
| **I（告知）** | 训练团队：如有数据延迟 |

### R6：元数据 schema 频繁变更

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 P1 |
| **说明** | 训练团队对配比维度需求变化导致元数据 schema 不断增删。每次 schema 变更可能需要重跑部分历史数据，导致工程返工和延迟 |
| **触发条件** | 训练团队提出新的配比维度需求（如新增 `difficulty_level` 字段） |
| **预警信号** | 单季度 schema 变更 > 3 次；历史数据重处理触发 > 1 次/月 |
| **缓解措施** | 1. 预留 `metadata_json` JSON 扩展字段，临时需求先存入扩展字段<br>2. 核心字段（source/lang/content_type/quality_score/hash）稳定不变<br>3. 变更流程：新需求 → 先在扩展字段试运行 1 个月 → 确认为长期需求后才升格为核心字段<br>4. Iceberg schema evolution 支持在线加列，不重写历史数据（新列为 NULL） |
| **R（执行）** | 数据团队：schema 设计与维护 + 扩展字段管理 |
| **A（负责）** | 数据链路负责人（你）：审批 schema 变更 |
| **C（咨询）** | 训练团队：配比需求和字段语义 |
| **I（告知）** | 平台团队：如 schema 变更影响下游系统 |

### R7：LSH 去重索引膨胀

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 P1 |
| **说明** | 每天 TB 增量，LSH 索引持续增长。随着数据量增加，LSH bucket 内候选项增多，单次 query 的候选集扫描成本增加，检索性能衰减 |
| **触发条件** | 增量数据累积超过 30 天未做 Compaction |
| **预警信号** | LSH 单次查询延迟 > 目标的 2 倍；LSH bucket 平均候选数 > 1000；日增量去重时间超过 6 小时 |
| **缓解措施** | 1. 定期 Compaction：微小（日）→ 中（周）→ 大（月），重建 LSH 索引<br>2. 冷数据归档：超过 90 天的数据从 LSH 索引中移除（冷数据不再参与去重）<br>3. 分片 LSH：按 source×lang 分片建独立 LSH 索引，降低单索引规模<br>4. 监控 LSH 索引大小和查询延迟，设立自动 Compaction 触发阈值 |
| **R（执行）** | 数据团队：Compaction 调度 + LSH 索引维护 + 性能监控 |
| **A（负责）** | 数据链路负责人（你） |
| **C（咨询）** | — |
| **I（告知）** | 训练团队：如去重延迟增加 |

### R8：合成数据质量差

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 P1 |
| **说明** | 内部模型生成的 instruction-response 对质量不达标：幻觉、格式错误、语义不连贯、与原始语料脱节。大量低质量合成数据进入训练集反而损害模型性能 |
| **触发条件** | 合成数据首次批量生成后进行人工抽检 |
| **预警信号** | 人工抽检通过率 < 70%；LLM Judge 评分方差 > 2σ（质量极不均匀）；合成数据与原始语料的语义相似度 < 0.5 |
| **缓解措施** | 1. 先小批量人工评估（500-1000 条），确认达标后再扩量<br>2. 模板生成打底（规则可控），LLM 生成做多样性增强<br>3. 双向验证：用不同模型互相验证生成内容的正确性<br>4. 对合成数据做与原始数据相同的评测流程（Perplexity + 质量模型 + LLM Judge）<br>5. 建立合成数据质量标准量规（Rubric）：准确性、相关性、多样性、格式规范性 |
| **R（执行）** | 数据团队：Prompt 工程 + 生成流程 + 质量验证 |
| **A（负责）** | 数据链路负责人（你）：拍板是否将合成数据加入训练集 |
| **C（咨询）** | 训练团队：合成数据的适用场景和比例上限 |
| **I（告知）** | 训练团队：合成数据的质量报告 |

## 4. P2 级风险

### R9：冷门语言 AST 覆盖不足

| 维度 | 内容 |
|---|---|
| **优先级** | 🟢 P2 |
| **说明** | Top-15 以外的编程语言（Haskell/Erlang/Solidity/Coq/Lean 等）缺乏 AST 解析能力。如果这些语言的高质量代码占比上升，结构化信息缺失会影响去重精度和代码质量评测 |
| **触发条件** | 冷门语言数据占比 > 5% 且对模型训练有重要价值 |
| **预警信号** | 冷门语言占比月增长 > 2%；训练团队提出特定语言的数据需求 |
| **缓解措施** | 1. 监控每个语言的占比和增长趋势<br>2. 定期（每季度）评估 Top-15 是否需要扩展<br>3. 冷门语言暂时降级为纯文本处理（规则过滤 + 文本去重）<br>4. 按需开发：训练团队提出需求时，评估 Tree-sitter parser 成熟度，投入 1-2 周开发 |
| **R（执行）** | 数据团队：语言分布监控 + 按需开发 AST |
| **A（负责）** | 数据链路负责人（你） |
| **C（咨询）** | 训练团队：是否需要某冷门语言的数据？ |
| **I（告知）** | — |

## 附录：RACI 角色定义

| 角色 | 全称 | 含义 |
|---|---|---|
| **R** | Responsible（执行） | 具体执行任务的人/团队 |
| **A** | Accountable（负责） | 对任务结果最终负责的人，拍板决策 |
