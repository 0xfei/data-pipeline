## 2. 高优先级待定项

### 待定项 1：Token 预计算策略

| 维度 | 内容 |
|---|---|
| **优先级** | 🔴 高 |
| **依赖方** | 训练团队 |
| **问题描述** | 数据链路是否需要提前 tokenize 再交付训练集，还是训练框架自己在线 tokenize？两种方案在存储量、GPU 利用率、tokenizer 迭代灵活性上有本质差异 |
| **当前立场** | Phase 1 不预 tokenize，推荐训练框架在线 tokenize |
| **决策要点** | 1. 训练团队使用的训练框架（Megatron/NeMo/FSDP）是否支持在线 tokenize + 预取流水线<br>2. tokenizer/BPE 词表在 Phase 1 期间是否可能迭代<br>3. 在线 tokenize 的 GPU 利用率损耗实测数据 |
| **两种方案对比** | 详见 [README.md §9.2](../README.md#92-token-预计算方案对比) |
| **下一步** | 与训练团队召开技术对齐会议，确认 tokenizer 稳定性和训练框架能力 |

### 待定项 2：Pipeline 编排框架最终选型

| 维度 | 内容 |
|---|---|
| **优先级** | 🔴 高 |
| **依赖方** | 内部平台评估 |
| **问题描述** | 数据链路需要 Pipeline 编排框架来调度多阶段（采集→过滤→加工→评测→格式转换）和跨引擎（Spark + Ray）的流水线。Dagster（推荐）和 DolphinScheduler（内部已有）各有利弊 |
| **当前立场** | Dagster 首选（Iceberg 原生 + Asset 模型），DolphinScheduler 备选（团队熟悉） |
| **决策要点** | 1. 内部 DolphinScheduler 是否已有成熟的 Spark + Ray 集成<br>2. 是否接受在 DolphinScheduler 上自建 Iceberg 集成（2-3 人月工作量）<br>3. 团队对引入新工具（Dagster）的学习成本和运维负担的承受能力 |
| **两种方案对比** | 详见 [README.md §9.1](../README.md#91-pipeline-编排框架对比) |
| **下一步** | 在 Phase 1 启动前完成 PoC：用 Dagster 和 DolphinScheduler 分别跑一个最小 Pipeline（采集→粗过滤→写入 Iceberg），对比开发体验和运维复杂度 |

## 3. 中优先级待定项

### 待定项 3：Tag 体系的具体维度和值

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 中 |
| **依赖方** | 训练团队 + 数据团队 |
| **问题描述** | 方案中预留了 ≤ 10 个 tag 的多标签多维度体系，但具体维度和枚举值尚未确定。Tag 体系决定了下游数据配比的灵活度 |
| **当前立场** | 多标签多维度：source / content_type / lang / quality_level 已确定为核心字段（不可为空）。额外 tag 维度和值后续与训练团队确定 |
| **建议维度和值**（供讨论） | 维度1-主题：`algorithms` / `systems` / `web_dev` / `data_science` / `mobile` / `devops` / `security` / `other`<br>维度2-难度：`beginner` / `intermediate` / `advanced`（需 LLM 分类）<br>维度3-格式：`tutorial` / `reference` / `discussion` / `qa`<br>维度4-时效性：`evergreen` / `versioned` / `archived` |
| **下一步** | 与训练团队讨论上述建议，收敛到 ≤ 10 个 tag。需确认哪些维度对配比有实际价值 |

### 待定项 4：元数据 schema 扩展维度

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 中 |
| **依赖方** | 训练团队 |
| **问题描述** | 训练团队可能提出更多配比维度需求（如数据难度、领域分类、token长度分层），需要在元数据 schema 中预留扩展空间 |
| **当前立场** | 核心字段稳定不变（source/lang/content_type/quality_score/hash），`metadata_json` JSON 扩展字段承载临时需求。确认为长期需求后再升格为核心字段 |
| **下一步** | 1. 向训练团队收集配比维度需求的完整清单<br>2. 评估哪些需求可以用现有字段组合实现（如 `source × quality_score` 已可表达大部分配比）<br>3. 仅对现有字段无法满足的需求，评估是否加入核心字段 |

### 待定项 5：标注团队方案

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 中 |
| **依赖方** | 数据团队 |
| **问题描述** | L4 人工评估（校准 LLM Judge）、合成数据质检、争议样本仲裁都需要标注人员。但目前团队未组建，具体人数、流程、工具待定 |
| **当前立场** | 暂定需要标注团队，但 Phase 1 可以先靠数据团队成员兼任（每月 200-500 条，工作量可控） |
| **待决策** | 1. 标注团队是内部全职还是兼职<br>2. 标注工具选型：Label Studio / Argilla / 自研<br>3. 标注标准的量规（Rubric）设计<br>4. 标注一致性（Kappa 系数）的达标标准 |
| **下一步** | Phase 1 先由数据团队兼职 2-3 人，Phase 2 再评估是否需要全职标注团队 |

### 待定项 6：License 过滤策略细节

| 维度 | 内容 |
|---|---|
| **优先级** | 🟡 中 |
| **依赖方** | 法务 |
| **问题描述** | GitHub 代码的 License 合规性是法律灰色地带。当前方案是「全量保留 + 法务兜底」，技术上提供 License 过滤能力。但具体哪些 License 需要过滤尚未明确 |
| **当前立场** | 技术上支持 License 检测和过滤（从 repo LICENSE 文件 + 文件头注释解析）。策略上等待法务给出明确的保留/过滤清单 |
| **待决策** | 1. 哪些 License 可以保留用于训练（MIT/Apache/BSD 等 permissive license 大概率安全）<br>2. 哪些 License 需要过滤（GPL/AGPL 等 copyleft license 可能有法律风险）<br>3. 无 License 的代码如何处理（默认保留 vs 默认过滤）<br>4. 多 License 混合的 repo 如何处理（如 MIT + GPL 双许可） |
| **下一步** | 整理 GitHub 数据中 License 分布报告，提交法务，等待正式意见。同时技术上准备好按 License 过滤的能力 |

## 4. 低优先级待定项

### 待定项 7：网页审美的具体实现路径

| 维度 | 内容 |
|---|---|
| **优先级** | 🟢 低 |
| **依赖方** | 训练团队 |
| **问题描述** | 「网页审美」作为文档质量代理信号的想法很新颖，但具体落地路径不明确。DOM 特征 vs 视觉模型 vs 第三方信号，投入产出比存疑 |
| **当前立场** | 作为 Phase 2 探索项，暂时保留。Phase 1 不实现 |
| **当前探索方向** | DOM 结构特征提取（文本/标签比、`<article>` 标签检测、代码块密度）作为低成本第一步 |
| **下一步** | 与训练团队确认：1) 是否需要这个信号？2) 预期解决什么问题（过滤低质量页面 vs 提升高质量文档的配比权重）？确认需求后再评估技术方案 |

### 待定项 8：冷门语言 AST 补充计划

| 维度 | 内容 |
|---|---|
| **优先级** | 🟢 低 |
| **依赖方** | 数据团队 |
| **问题描述** | Top-15 以外的编程语言暂时只做纯文本处理。如果某冷门语言的高质量代码占比提升，或者训练团队提出特定语言需求，需要补充 AST 支持 |
| **当前立场** | 目前 Top-15 语言覆盖 95%+ 代码量，冷门语言降级为纯文本处理。后续按需补充 |
| **可能触发补充的语言** | Haskell、Erlang、Solidity（智能合约）、Coq/Lean（形式化验证）、R（数据科学）、Dart（Flutter）、Elixir |
| **下一步** | 1. 监控每个语言的月度占比和增长率<br>2. 每季度评估是否需要扩展 Top-N<br>3. 如训练团队提出需求 + Tree-sitter parser 成熟，1-2 周内补充 |
