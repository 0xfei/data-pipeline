## 1. 概述

本文档面向**数据团队**，详细讲解 70B 参数大模型从数据交付到线上推理的完整训练流程。理解训练过程能帮助数据团队更好地提供数据支持，理解数据配比、token 预估、checkpoint 回溯等关键环节的价值。

**核心脉络**：

```
数据交付 → Tokenize → 分布式训练 → Checkpoint → 评估 → 量化 → 上线推理
```

## 2. 从数据到 Token IDs

### 2.1 数据如何进入训练管线

这是训练管线与数据链路最直接的接口：

```
JSONL 文件 (你交付的)
  ↓
Tokenizer（分词器，把文本变成数字序列）
  ↓
Token IDs：[15496, 2159, 382, 11994, ...]
  ↓
Batching（打包成 batch，按序列长度分组减少 padding 浪费）
  ↓
送入 GPU 计算
```

#### Batch 构建策略

序列长度直接影响 GPU 利用率和训练质量：

| 参数 | 典型值（70B） | 影响 |
|---|---|---|
| `max_seq_len` | 4096 / 8192 / 32768 | 越长越吃显存（注意力矩阵 O(n²)），但能学更长的上下文 |
| `global_batch_size` | 4M-8M tokens/batch | 越大训练越稳定，但需要更多 GPU |
| `micro_batch_size` | 2-8 per GPU | 受单卡显存限制 |

**长度分桶（Bucketing）**：将相近长度的序列放在同一个 batch 中，减少 padding 浪费：

```
不分桶:
  batch: [1024 tokens, 4096 tokens, 128 tokens, 2048 tokens]
  所有序列 padding 到 4096 → 浪费 60% 计算量

分桶后:
  桶1 (0-512):   [128, 256, 384, 512]   → padding 到 512，浪费少
  桶2 (513-1024): [768, 896, 1024, 640]  → padding 到 1024
  桶3 (1025-4096): [2048, 3072, 4096, 1536] → padding 到 4096
```

**对数据链路的影响**：如果数据中有大量短文本（< 256 tokens）和少量长文本（> 4096 tokens），GPU 大量时间花在 padding 上，有效计算率低。**数据团队应尽量让文本长度分布均匀，或帮助训练团队做长度分桶预处理。**

### 2.2 Tokenizer 是什么

把 tokenizer 想象成一本**密码本**——把文字翻译成数字：

```
"我"  → 105
"喜欢" → 28976
"编程" → 4521
"def"  → 1823
```

**关键概念**：

- **预训练阶段** tokenizer 通常在整个训练过程中**固定不变**（因为换了 tokenizer，embedding 层的维度就变了，模型得从零重新训练）
- 多轮训练（v1 → v2）之间**可能换 tokenizer**（扩词表、优化分词策略）——这就是为什么数据链路不预 tokenize
- 词表大小通常在 50K-250K 之间，中文大模型通常更大

### 2.3 多语言 + 多编程语言的 Tokenizer 设计

当前主流方案是**单 Tokenizer 处理全语言**。使用 BPE（Byte-Pair Encoding）算法在混合语料上训练。

#### BPE 基本原理

```
原始文本: "def hello_world():"

Step 0: 按字符拆分 → ['d','e','f',' ','h','e','l','l','o','_','w','o','r','l','d','(',')',':']

Step 1: 合并最高频的相邻对 → ['de','f',' ','he','l','l','o','_','w','o','r','l','d','(',')',':']

...

Step N: 最终 → ['def', 'hello', '_', 'world', '():']
```

#### 词表在多语言间的自然分配

经过在混合语料（中+英+代码）上训练后，词表会自然形成：

```
约 100K tokens 的词表分配:

  英文常见词/子词    ████████████████████████████████  40%
  代码模式           ██████████████████████████        25%
  中文常见字/词      ██████████████████████            20%
  其他语言           ██████████                        10%
  特殊 token         ███                               5%  (如 <|endoftext|>, <|user|>)
```

#### 不同语言的分词效率差异

| 语言/内容 | 分词效率 | 示例 | 说明 |
|---|---|---|---|
| **英文** | 极高（~1.2 tokens/词） | "hello world" → [hello, world] 2 tokens | 同等语义密度下 token 最少 |
| **中文** | 中等（~1.5-2.5 tokens/字） | "你好世界" → [你, 好, 世界] 3 tokens | 常见字一个 token，生僻字被拆成多个 byte token |
| **代码（Python）** | 高（~1.3 tokens/词） | "def hello():" → [def, hello, ():] 3 tokens | 常见关键字/模式优先分配短 token |
| **代码（Rust/长标识符）** | 中低 | "get_async_runtime_handle" → [get, _, async, _, runtime, _, handle] 7 tokens | 长标识符被拆成多个 token |
| **日/韩文** | 低（~3-5 tokens/字） | BPE 训练不充分的语言更碎片化 | 罕见语言数据产生大量 token 但信息密度低 |

#### 关键启示：配比不能只看 GB，要看 Token 预估

```
同样的内容量（按 GB 计算），不同语言产生的 token 数差异巨大：

100GB 英文  ≈ 25B tokens   ██████████████
100GB 中文  ≈ 35B tokens   ██████████████████  ← 多 40%
100GB 代码  ≈ 28B tokens   ████████████████
100GB 其他  ≈ 40B tokens   ████████████████████  ← 碎片化严重
```

**如果按 GB 数 1:1 配比中英文，实际中文 token 占比会远超英文，导致模型对中文分布过拟合。**

### 2.4 换 Tokenizer 的三种情况

#### 情况 1：扩词表（不重新训练，只做增量迁移）✅ 最常见

```
场景：v1 训练完后，发现中文/代码 token 太少，想加几千个新 token

做法：
  1. 在 v1 tokenizer 的词表上追加新 token（如 +5000 个中文/代码 token）
  2. 模型的 embedding 层从 50K×4096 扩展到 55K×4096
  3. 新增的 5000×4096 参数用随机初始化
  4. 已有 50000×4096 参数从 checkpoint v1 直接拷贝
  
结果：
  - 不需要从零重新训练
  - 只需要少量（几十 B tokens）的 continue training 让新 token "预热"
  - 这是行业标准做法（LLaMA → Chinese-LLaMA, Qwen 扩词表等）
```

#### 情况 2：换分词算法（必须重新训练）❌ 极少发生

```
场景：从 BPE 换到 SentencePiece 或 Unigram

影响：
  - 分词方式完全不同，同一个句子会被切成完全不同的 token 序列
  - v1 的 embedding 映射完全失效
  - 必须从零训练
```

#### 情况 3：换迭代版本（通常不重新训练）⚠️

```
场景：训练 v1 时用的是 tokenizer v1，训练 v2 时想用 tokenizer v2

分析：
  - 如果只是 BPE merge 规则的差异（同算法）：大多数高频 token 的 ID 不会变
  - 但总有一部分 token 的 ID 会不同 → 旧数据的 token ID 序列会部分错乱
  
启示：如果数据链路存的是原始文本，换个 tokenizer 只是训练框架调用函数变了，零成本
```

#### 换 Tokenizer 的常见场景总结

| 场景 | 原因 | 频率 | 处理方式 |
|---|---|---|---|
| **多语言支持不足** | 原 tokenizer 主要在英文上训练，中文/日文碎片化严重 | 很常见 | 扩词表（情况 1） |
| **代码 token 效率低** | 缩进/空格占大量 token；长标识符被拆解 | 常见 | 扩词表，加代码专用 token |
| **特殊领域适配** | 数学公式、化学式、法律条文有特殊符号 | 偶尔 | 扩词表 |
| **升级基座 tokenizer** | 换更强的基座模型（如 LLaMA→LLaMA2 tokenizer） | 很少 | 从零训练或做外科手术级权重迁移 |
| **降低词表大小** | 显存不够，剪枝低频 token | 极少 | 需裁剪 embedding 层，需重训 |

### 2.5 数据团队的数据量有多少会进入训练

```
100PB 原始数据 → 最终 1-2T tokens → ~4-8TB 训练文本

过滤链路：
  100PB 原始数据
  → 粗过滤后 40-60PB（SHA256 + AST + PII 脱敏）
  → 粗加工后 30-50PB（分类/标签/重要性）
  → MinHash 去重后 20-35PB
  → 评测漏斗后 5-15PB
  → 训练团队配比后 1-2T tokens ≈ 4-8TB 文本
```

| 指标 | 量级 | 说明 |
|---|---|---|
| 原始数据总量 | 100PB | 采集入库的原始内容 |
| 最终训练量 | 1-2T tokens | 约模型参数量 × 20（Chinchilla 定律） |
| 最终文本量 | ~4-8TB | 1T tokens ≈ 4TB 英文文本 |
| 总过滤率 | ~98-99% | 100PB 中只有 1-2% 最终进入训练 |
| 单次 epoch 时间 | 3-7 天 | 1024 卡 H100，1-2T tokens |
| 总训练 epoch | 1-3 个 | 数据量大的模型通常只训 1 个 epoch |

> **核心认知**：你不是在"生产 100PB 训练数据"。你是在"从 100PB 矿渣中筛选出 4-8TB 高纯度矿石"。过滤率 98-99% 是正常的，你的工作价值在于那 1-2%。

## 3. 分布式并行训练

### 3.1 为什么需要分布式训练

```
70B × BF16 (2 bytes) = 140GB 纯权重
+ Adam 优化器状态 = ~700GB-1.2TB 一个完整的训练状态
单张 A100-80GB: 连权重的 1/8 都装不下

解决方案：把模型"切碎"，分布在 1024 张 GPU 上协同计算
```

### 3.2 四种并行策略

```
一个 70B 模型 = 80 层 Transformer × 8192 维度

┌─────────────────────────────────────────────────────┐
│  数据并行 (DP)                                       │
│  每 8 张 GPU 一组，128 组并行处理不同的数据 batch     │
│  GPU 0-7: batch 1    GPU 8-15: batch 2              │
│  每组内还需继续切分...                                │
├─────────────────────────────────────────────────────┤
│  张量并行 (TP)                                       │
│  同一层 Transformer 被横着切开                       │
│  [8192×8192 矩阵] = GPU0 [8192×2048] + GPU1 [...]   │
│  每 4 张 GPU 分一层                                 │
├─────────────────────────────────────────────────────┤
│  流水线并行 (PP)                                     │
│  80 层 Transformer 按层切分                          │
│  GPU0: 层 1-20 → GPU1: 层 21-40 → GPU2: 层 41-60   │
│  像一个流水线工厂                                     │
├─────────────────────────────────────────────────────┤
│  ZeRO (内存优化)                                     │
│  优化器状态分片存储，省下 80% 显存                     │
└─────────────────────────────────────────────────────┘
```

#### 典型 70B 模型的并行配置

```
TP=4, PP=2, DP=128
4 × 2 × 128 = 1024 GPU
```

每个 DP 组内的 8 张 GPU（4 TP × 2 PP）接收**完全相同**的 token ID batch，通过张量并行切分和流水线并行分层，共同完成一份数据的计算。128 个 DP 组各自处理不同的数据。

### 3.3 Token IDs 在 GPU 之间的旅程

#### Step 1: 数据加载（CPU 侧）

```
128 个 DP 组各自独立读取不同的 JSONL 文件:

DP组 0:  读取 part-00001 ~ part-00008.jsonl
DP组 1:  读取 part-00009 ~ part-00016.jsonl
...
DP组 127: 读取 part-01017 ~ part-01024.jsonl

⚠️ 每轮 epoch 结束后 shuffle，避免同一组永远读同一段数据
```

#### Step 2: Tokenize（CPU 侧）

```
每个 DataLoader 调用 tokenizer:
  "def hello_world():"  →  [15496, 2159, 3598, 7, 382]
  "今天天气真好"         →  [10423, 10165, 10189, 9934, 10045]

然后打包成 batch（按长度分桶、padding 短序列为 0）
```

#### Step 3: 入 GPU（CPU → GPU，PCIe）

```
每个 DP 组内的 8 张 GPU 接收完全相同的 token ID batch
(因为 TP=4、PP=2，这 8 张卡共同算同一份数据)

Token IDs → Embedding 层 → 稠密向量 [bsz, 4096]
```

#### Step 4: 前向传播（GPU 间通信）

```
GPU 0-3 (PP 阶段 0: 层 1-40):
  每层计算时 GPU 之间通过 NCCL 通信:
    Attention: all-reduce 各卡的部分结果
    FFN:      各算各的（已切分好）

  层 40 输出 → 跨 GPU 传给 GPU 4-7

GPU 4-7 (PP 阶段 1: 层 41-80):
  接收 GPU 0-3 的输出 → 继续计算 → 输出 logits
```

#### Step 5: 反向传播（GPU 间通信 + 跨 DP 组同步）

```
GPU 4-7: 算层 80→41 的梯度 → 传给 GPU 0-3
GPU 0-3: 算层 40→1 的梯度（含 embedding 层）

128 个 DP 组各自算出了梯度 → 全部做 all-reduce 求平均
→ 每组得到相同的平均梯度 → 更新参数 → 一个 step 完成
→ 返回 Step 3，加载下一批 token IDs
```

#### 数据传输汇总

| 传输的是什么 | 什么时候传 | 传多大 | 通道 |
|---|---|---|---|
| Token IDs | 前向传播开始 | 几 MB/batch | PCIe |
| 中间激活值 | PP 的层间传递 | 几十 MB | InfiniBand |
| TP 的 all-reduce | 每层 Attention 后 | 几十 MB | NVLink |
| 梯度同步 | 反向传播后，跨 DP 组 | ~1GB/DP 组 | InfiniBand |
| 优化器状态 | 参数更新时 | 本地更新，不传 | — |

#### CPU 和内存的角色

```
每张 GPU 前面的 "后勤部队":

┌─────────────────────────────────────────┐
│           CPU (4-8 核心/GPU)             │
│  负责:                                   │
│  1. 从 S3 读 JSONL 文件                  │
│  2. 解压缩 (ZSTD/Snappy)                 │
│  3. Tokenize (文本 → token IDs)          │
│  4. 打包 batch (padding + mask)          │
│  5. 预取下一批数据                         │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────┴───────────────────────┐
│           RAM (16-32GB/GPU)              │
│  预取 2-3 个 batch 的 token ID 数组       │
│  避免 GPU 等 S3 读取 + tokenize           │
└─────────────────┬───────────────────────┘
                  │ PCIe
┌─────────────────┴───────────────────────┐
│           GPU (A100-80GB)                │
│  负责:                                   │
│  Embedding lookup + Transformer + Loss   │
└─────────────────────────────────────────┘

黄金配比: 每张 GPU 配 4-8 CPU 核心 + 16-32GB 缓冲区
```

### 3.4 GPU 调度体系

**Ray 不是训练 GPU 的调度器**——训练和数据处理走两套完全不同的体系：

| 场景 | 调度框架 | 说明 |
|---|---|---|
| **数据链路**（Embedding、评测 L2） | **Ray** | Python 生态、灵活批处理 |
| **模型训练** | **torchrun + NCCL + DeepSpeed/Megatron** | 毫秒级同步、大规模 all-reduce |

#### 为什么训练不用 Ray？

```
训练的特点：
  - 1024 张 GPU 必须同步做 all-reduce
  - 调度延迟要求微秒级
  - 需要直接控制 NCCL 通信拓扑

Ray 的特点：
  - 异步任务调度，延迟毫秒级
  - 不擅长 1024 张 GPU 的同步屏障 (barrier)

如果用 Ray 调度训练 → 调度开销吃掉 10-20% 性能
```

#### 训练 GPU 调度栈

```
Slurm / K8s+Volcano (集群资源管理)
  ↓
torchrun / mpirun (进程启动: rank 0-1023)
  ↓
PyTorch Distributed + NCCL (通信: all-reduce)
  ↓
DeepSpeed / Megatron-LM / FSDP (训练框架: TP/PP/DP)
```

## 4. Checkpoint 与模型评估

### 4.1 Checkpoint 是什么

Checkpoint 是**某一时刻模型全部参数的快照**——训练过程中的"存档点"。

```
没有 checkpoint:
  训到第 25 天 → GPU 内存错误 → 💥 → 从零重训 25 天

有 checkpoint:
  训到第 25 天 → 💥 → 从 checkpoint_50000 (第 24 天) 恢复 → 只丢 1 天
```

#### Checkpoint 的内容

```
一份完整 checkpoint ≈ 700GB-1.3TB

┌─────────────────────────────────────────────┐
│  model_weights/          (~140 GB)          │
│  ├── embedding.weight    (词表×维度)         │
│  ├── layer_0...79        (80层Transformer)   │
│  └── lm_head.weight      (输出层)            │
│                                             │
│  optimizer_states/      (~560 GB-1.12 TB)   │
│  ├── exp_avg (momentum)                     │
│  └── exp_avg_sq (variance)                  │
│                                             │
│  training_state.json     (~1 KB)            │
│  └── {"step": 50000, "epoch": 1.2,         │
│        "loss": 2.345, "tokens_consumed":    │
│         "500B", "dataset_version":          │
│         "gold_v1@snap_20260615"}            │
└─────────────────────────────────────────────┘
```

#### Checkpoint 保存流程

```
每 1000-2000 步自动触发:

1. 1024 张 GPU 暂停计算 (barrier)
2. 每张 GPU 把参数从显存 → CPU 内存
3. CPU 内存 → 本地 NVMe SSD (10-30 秒)
4. 本地 SSD → S3 (异步，不阻塞，5-15 分钟)

频率权衡:
  每 500 步:  安全但浪费 5% 训练时间
  每 2000 步: 高效但最多丢 2000 步
  业界惯例: 每 1000-2000 步，每天一次永久保留
```

#### Checkpoint 与数据链路的关联

```
全链路追溯:

checkpoint_50000
  ↓ (读 training_state.json → dataset_version)
gold_pretrain_v1@snap_20260615
  ↓ (读 pipeline_executions 表)
Iceberg snapshot #7520426094023953954
  ↓
bronze snapshot #6230415083912842034
  ↓
raw snapshot #5129384062811730923
  ↓
原始 URL: https://github.com/xxx/yyy/blob/main/zzz.py
采集时间: 2026-05-20 03:15:22 UTC
处理参数: 去重阈值=0.8, embedding=v1.2
```

**如果 checkpoint_50000 的 loss 异常，数据团队能在 10 分钟内定位到问题数据的原始 URL。**

### 4.2 模型评估

## 5. 从训练到上线

### 5.1 格式转换与量化

#### 训练 Checkpoint 为什么不能直接用？

```
训练时的 checkpoint (700GB-1.3TB):
  ├── model_weights (BF16)        140 GB  ✅ 推理需要
  ├── optimizer_states            ~800 GB  ❌ 推理不需要
  └── 其他训练状态                 几 MB   ❌ 推理不需要

推理只需要 140GB 权重。而且 140GB > 80GB → 单卡装不下 → 需要量化
```

#### 量化（模型的"有损压缩"）

| 量化精度 | 显存 (70B) | 质量损失 | 适用场景 |
|---|---|---|---|
| **BF16** (原始) | 140 GB | 0% | 训练 / 研究 |
| **FP8** | 70 GB | < 1% | A100/H100 推理 |
| **INT4** | 35 GB | 1-3% | 低成本部署 |
| **INT4 + GPTQ/AWQ** | 35 GB | < 1% | 高性价比推理 |

```
量化后:
  BF16: ████████████████████████████████████████  140 GB (4×A100)
  INT8: ██████████████████                        70 GB (2×A100)
  INT4: █████████                                   35 GB (1×A100)
```

**关键启示**：高质量数据训练的模型 → INT4 量化后损失 < 1%；低质量数据训练的模型 → INT4 量化后损失可能 5-10%。**数据质量直接决定推理成本。**

#### 格式转换工具

| 工具 | 输出格式 | 用途 |
|---|---|---|
| HuggingFace Transformers | safetensors | 最常用，生态丰富 |
| vLLM | GPU 内存格式 | 高并发在线推理 |
| TensorRT-LLM | TensorRT Engine | NVIDIA 官方极致优化 |
| llama.cpp | GGUF | CPU 推理 / 本地部署 |

### 5.2 线上推理架构

```
                    用户请求
                  "用 Python 写个快排"
                       │
                       ▼
               ┌──────────────┐
               │   API 网关    │  (限流/鉴权/路由)
               └──────┬───────┘
                      │
         ┌────────────┼────────────┐
         ▼            ▼            ▼
   ┌──────────┐ ┌──────────┐ ┌──────────┐
   │推理实例 0 │ │推理实例 1 │ │推理实例 2 │
   │ GPU 0    │ │ GPU 1    │ │ GPU 2    │
   └──────────┘ └──────────┘ └──────────┘
         │            │            │
         └────────────┴────────────┘
                      │
                      ▼
              返回 token 流
        "def quicksort(arr):" ...
```

#### 两个关键技术

**1. KV Cache（注意力机制的缓存）**

```
每次生成新 token 时，不用重算所有历史 token 的 attention
→ 无论序列多长，每步计算量基本恒定
→ 显存消耗随序列长度线性增长

70B 模型 KV Cache (4096 序列，FP16):
  ≈ 2 × 80层 × 4096 × 8192 × 2 bytes ≈ 10 GB

量化模型 35GB + KV Cache 10GB = 45GB
→ 一张 A100-80GB 刚好够用
```

**2. Continuous Batching（动态批处理）**

```
传统 batching:
  请求1 只生成 50 tokens 就完了 → GPU 空等请求 2 和 3

vLLM Continuous Batching:
  请求1 生成完 → 立刻补充新请求 → GPU 几乎 100% 利用
  一个 batch 里混合不同阶段的请求
```

#### 推理框架对比

| 框架 | 优势 | 劣势 |
|---|---|---|
| **vLLM** | PagedAttention（极省显存）、吞吐极高 | 仅 NVIDIA GPU |
| **TensorRT-LLM** | NVIDIA 官方极致优化 | 部署复杂 |
| **SGLang** | 前缀缓存、结构化输出 | 相对新 |
| **llama.cpp** | CPU 推理、GGUF 格式 | 速度慢 |

## 7. 对数据团队的关键启示

### 7.1 数据格式

- ✅ 存原始文本（JSONL），不预 tokenize
- ✅ Tokenizer 变更时，原始文本零成本切换
- ✅ 支持长度分桶预处理，帮助训练团队提升 GPU 利用率

### 7.2 数据配比

- ⚠️ 配比不能只看 GB，要看 token 预估（中英文 token 膨胀率差异 40%+）
- ⚠️ 数据 shuffle 必须全局化，避免同一 DP 组永远读同一分布的数据
- ⚠️ 验证集必须和训练集无重叠，否则 validation loss 虚低

### 7.3 数据质量

- ⚠️ 高质量数据 → INT4 量化后损失 < 1%
- ⚠️ 低质量数据 → INT4 量化后损失可能 5-10%
- ⚠️ 数据质量直接决定推理成本

### 7.4 Checkpoint 回溯

- ✅ 训练状态中的 dataset_version 是数据团队和训练团队的关键接口
- ✅ 全链路回溯能力是排查训练异常的杀手锏

### 7.5 推理反馈闭环

## 附录 A：分布式训练深度技术

> 以下为深度技术内容，供感兴趣的数据团队成员参考。不影响对主文档的理解。

### A.1 TP（张量并行）的数学本质

```
一个 Transformer 层的核心计算是矩阵乘法 Y = X × W

X: [batch_size, d_model]  如 [8, 8192]
W: [d_model, d_model]     如 [8192, 8192]

TP 切法:
  W 按列切成 4 份: W = [W₀, W₁, W₂, W₃]
  每张 GPU 算自己的那部分: Yᵢ = X × Wᵢ

前向: X 广播到所有 GPU → 各自算 Yᵢ → concat 得到完整 Y
反向: Y 的梯度广播 → 各自算 Wᵢ 的梯度 → 更新 Wᵢ
```

### A.2 ZeRO 的三个阶段

| 阶段 | 分片内容 | 显存节省 |
|---|---|---|
| ZeRO-1 | 优化器状态 | 4× |
| ZeRO-2 | 优化器状态 + 梯度 | 8× |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | 与 GPU 数线性扩展 |

### A.3 混合精度训练

```
前向传播: BF16 (快，省显存)
反向传播: BF16 (梯度计算)
参数更新: FP32 (精度高，避免舍入误差)

Master Weight: FP32 (存在 CPU 或特定 GPU 上)
Working Weight: BF16 (存在所有 GPU 上)
```
